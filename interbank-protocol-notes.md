## OTC Negotiation â€” OtcOffer

Sent on `POST /negotiations` (create) and `PUT /negotiations/{rn}/{id}` (counter-offer).

Routing number is sellers routing number and ID is same ID used on both sides to identify negotiation generated by seller on initial offer

### Extension: `buyerAccountNumber`

Not in the base spec. Agreed extension across all banks in this project.

The buyer must declare a specific account for settlement upfront. We use it to debit the premium on accept and the strike on exercise. Without it there is no way to know which account to charge at transaction time.

- Required on `POST` â€” missing = 400
- Immutable â€” cannot change on `PUT`, returns 400
- Immutable alongside: `stock.ticker`, `buyerId`, `sellerId`

### Full wire shape

```json
{
  "stock":              { "ticker": "CBSH" },
  "settlementDate":     "2027-06-01T00:00:00Z",
  "pricePerUnit":       { "currency": "USD", "amount": 44 },
  "premium":            { "currency": "USD", "amount": 2 },
  "buyerId":            { "routingNumber": 111, "id": "some-buyer-id" },
  "sellerId":           { "routingNumber": 444, "id": "10" },
  "amount":             10,
  "lastModifiedBy":     { "routingNumber": 111, "id": "some-buyer-id" },
  "buyerAccountNumber": "111000165540581321"
}
```

### Turn rule

Buyer's turn when `lastModifiedBy.routingNumber â‰  buyerId.routingNumber`. Acting out of turn returns 409.

Counter-offer `PUT` must include all fields (including immutable ones echoed back unchanged). Returns 204.

## Accept flow â€” NEW_TX

Triggered by `GET /negotiations/{sellerRn}/{id}/accept`. Moves the premium and forms the option contract.

### Forwarding

The accepting bank **never** forms the transaction itself. When a party accepts the current offer, its bank sends `GET /negotiations/{sellerRn}/{id}/accept` to the **other party's bank** (the counterparty), and that receiving bank forms the NEW_TX below.

- Buyer accepts â†’ buyer's bank calls the **seller's** bank â†’ seller's bank forms the TX.
- Seller accepts â†’ seller's bank calls the **buyer's** bank â†’ buyer's bank forms the TX.

`{sellerRn}/{id}` in the path is **always the seller's routing number and the negotiation id the seller generated on creation**. It identifies the negotiation and is shared unchanged by both banks â€” regardless of who is accepting or which bank receives the request. It is *not* the destination: the request is always sent to the counterparty, while the path keeps the seller's `{rn}/{id}` so both sides resolve the same negotiation.

**Sign:** `amount < 0` = asset leaves (credit out), `amount > 0` = asset enters (debit in).

| # | Account | Amount | Asset | Why this account type |
|---|---|---|---|---|
| 1 | `ACCOUNT` â†’ `buyerAccountNumber` | `âˆ’premium` | `MONAS` | We use account specified by buyer in the offer |
| 2 | `PERSON` â†’ `sellerId` | `+premium` | `MONAS` | Seller's bank will credit whichever account they choose for that identity |
| 3 | `PERSON` â†’ `sellerId` | `âˆ’1` | `OPTION` | Seller gives the option right |
| 4 | `PERSON` â†’ `buyerId` | `+1` | `OPTION` | Buyer receives the option right |

## Exercise flow â€” NEW_TX

Triggered by `POST /api/peer-otc/contracts/{sellerRn}/{id}/exercise` from the buyer.

| # | Account | Amount | Asset | Why this account type |
|---|---|---|---|---|
| 1 | `ACCOUNT` â†’ `buyerAccountNumber` | `âˆ’(price Ã— qty)` | `MONAS` | Same settlement account declared in the offer |
| 2 | `OPTION` â†’ `contractId` | `+(price Ã— qty)` | `MONAS` | The seller's bank will find account connected with contract to credit to |
| 3 | `OPTION` â†’ `contractId` | `âˆ’qty` | `STOCK` | Shares are taken from the contract, which holds the seller's share reservation |
| 4 | `PERSON` â†’ `buyerId` | `+qty` | `STOCK` | Buyer receives the shares |

The `OPTION` account is a pseudo-account representing the contract itself. It differs from the OPTION asset in the accept flow:
- Accept: `OPTION` as **asset** â€” describes *what right* is being transferred
- Exercise: `OPTION` as **account** â€” the contract acts as escrow; funds and shares flow through it
